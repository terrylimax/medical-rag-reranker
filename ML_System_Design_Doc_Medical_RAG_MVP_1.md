# ML System Design Doc - [RU]

## Дизайн ML системы - Medical RAG Assistant MVP 1

_Заполнено по шаблону Reliable ML для ВКР-проекта `medical-rag-reranker`. В рамках ВКР роли `Product Owner` и `Data Scientist` совмещены одним исполнителем, поэтому бизнесовые и технические решения описаны в одном документе._

### 1. Цели и предпосылки

#### 1.1. Зачем идем в разработку продукта?

- Бизнес-цель:
  - создать MVP медицинской question-answering системы, которая сокращает время поиска релевантной медицинской информации по открытым авторитетным источникам и выдает ответ вместе с источниками.
  - Врачи быстрее ищут ответы для сложных случаев анамнеза, пациенты ознакамливаются с релевантной именно их запросу информацией, без диагноза.
- Почему станет лучше, чем сейчас:
  - ручной поиск по NIH/PubMed требует много времени и возвращает много нерелевантных документов;
  - простой keyword search не умеет хорошо ранжировать длинные медицинские ответы;
  - RAG с нейросетевым реранкером должен повышать релевантность top-документов и удобство получения ответа.
- Что считаем успехом итерации с точки зрения бизнеса:
  - есть воспроизводимый baseline и MVP-кандидат;
  - retrieval-качество и качество ответа измеряются на фиксированном benchmark;
  - система работает как демонстрационный API/MVP-сервис;
  - ответ сопровождается ссылками на использованные документы.

#### 1.2. Бизнес-требования и ограничения

- Краткое описание требований:
  - система должна отвечать на медицинские вопросы на основе открытых источников;
  - на первом этапе основной корпус строится из MedQuAD/NIH QA, дополнительные источники: PubMed и StatPearls;
  - ответ должен быть кратким, понятным и сопровождаться источниками;
  - решение должно быть воспроизводимым и пригодным для демонстрации в виде API/MVP.
- Бизнес-ограничения:
  - результат вывода системы не является медицинской рекомендацией и не должен использоваться для автономной постановки диагноза; только информационное ознакомление
  - в рамках MVP не интегрируется с EHR/EMR, внутренними медицинскими системами и персональными данными пациентов;
  - вычислительные ресурсы ограничены локальным или single-host окружением;
  - приоритет первой итерации: качество retrieval и groundedness, а не полнота production-функций.
- Что ожидаем от конкретной итерации:
  - подготовить очищенный корпус и фиксированные сплиты;
  - собрать baseline RAG без реранкера;
  - обучить и оценить нейросетевой реранкер;
  - подготовить MVP-сервис с API и контейнеризацией.
- Описание бизнес-процесса пилота:
  - пользователь задает медицинский вопрос через API или простой UI;
  - система ищет релевантные документы, ранжирует их и формирует ответ;
  - пользователь видит ответ и список источников;
  - пользователь использует результат как информационную справку, а не как самостоятельное клиническое решение.
- Что считаем успешным пилотом:
  - реранкер улучшает retrieval-метрики относительно baseline;
  - ответы опираются на retrieved-контекст и содержат корректные ссылки на источники;
  - сервис стабильно отрабатывает демонстрационный сценарий с приемлемой задержкой;
  - по итогам пилота понятен следующий шаг: доработка до production-like версии либо ограничение решения demo-форматом.

#### 1.3. Что входит в скоуп проекта/итерации, что не входит

- Входит в скоуп:
  - подготовка и очистка корпуса MedQuAD/NIH QA;
  - формирование `qa.jsonl`, `corpus.jsonl`, `splits.json`, `eval_queries.jsonl`, `qrels.tsv`;
  - baseline retrieval на BM25;
  - эксперименты с dense и hybrid retrieval;
  - обучение cross-encoder реранкера на PubMedBERT или близкой biomedical модели;
  - generation-слой для ответа по top-документам;
  - офлайн benchmark и ручной error analysis;
  - MVP API-сервис в Docker Compose.
- Не входит в скоуп:
  - полноценная клиническая валидация на реальных пациентах;
  - поддержка многоязычности;
  - интеграция с информационными системами медицинских организаций;
  - SRE-процессы промышленного уровня, высокая доступность, развертывание в нескольких регионах;
  - сертификация как медицинского ПО.
- Результат с точки зрения качества кода и воспроизводимости:
  - единый Python-проект с конфигами Hydra;
  - зависимости управляются через Poetry;
  - данные и артефакты организованы через DVC/local storage/AWS S3;
  - эксперименты логируются в MLflow;
  - сервисный контур разворачивается через Docker/Docker Compose.
- Планируемый технический долг:
  - полноценная валидация цитат и post-processing ответа;
  - online-monitoring качества ответов;
  - guardrails для небезопасных запросов;
  - UI для конечного пользователя;
  - hard-negative mining на основе BM25/hybrid;
  - production-ready dense/hybrid контур с фиксированными зависимостями и артефактами.

#### 1.4. Предпосылки решения

- Первая итерация ориентирована на английский язык, потому что основные корпуса и biomedical модели англоязычные.
- Основная сущность retrieval-контура:
  - запрос пользователя;
  - документ-кандидат;
  - score retriever;
  - score reranker;
  - итоговый ответ с ссылками на документы.
- В baseline документом считается ответ из QA-пары. Это упрощает построение benchmark и делает задачу воспроизводимой.
- В основном MVP документы расширяются за счет PubMed и StatPearls, что приближает систему к реальному RAG-сценарию.
- Качество first-stage retrieval критично, потому что генератор и реранкер могут работать только с теми кандидатами, которые попали в изначальные top-k.
- Первая оценка проводится офлайн на отложенном тестовом бенчмарке. Только после этого имеет смысл ограниченный экспертный пилот.
- Из-за ограниченных вычислительных ресурсов inference организуется асинхронно через job-based API.

### 2. Методология

#### 2.1. Постановка задачи

- С технической точки зрения решение состоит из трех связанных ML/IR-задач:
  - retrieval: найти top-k релевантных документов по медицинскому вопросу;
  - reranking: переупорядочить найденные кандидаты нейросетевой моделью cross-encoder;
  - answer generation: сформировать обоснованный ответ по top-документам с цитированием источников.
- Главная ML-задача итерации: learning-to-rank для medical retrieval.
- Генерация ответа в MVP вторична относительно retrieval/reranking и служит демонстрацией прикладной ценности решения.

#### 2.3. Этапы решения задачи

#### 2.3.1. Этап 1. Подготовка данных

Цель этапа: получить воспроизводимый benchmark и артефакты, на которых можно честно сравнивать baseline и MVP без утечек данных.

| Название данных                    | Есть ли данные                          | Требуемый ресурс для получения данных | Проверено ли качество данных     |
| ---------------------------------- | --------------------------------------- | ------------------------------------- | -------------------------------- |
| MedQuAD / NIH QA пары вопрос-ответ | Да, `data/raw/medquad/train.parquet`    | DS                                    | Да, базовая очистка и фильтрация |
| PubMed abstracts                   | Планируется через `MedRAG/pubmed`       | DS, доступ к HF datasets              | Частично, требуется EDA          |
| StatPearls passages                | Планируется через `MedRAG/statpearls`   | DS, доступ к HF datasets              | Нет, требуется EDA               |
| Производный QA benchmark           | Да, `data/processed/qa.jsonl`           | DS                                    | Да                               |
| Производный retrieval corpus       | Да, `data/processed/corpus.jsonl`       | DS                                    | Да                               |
| Сплиты train/val/test              | Да, `data/processed/splits.json`        | DS                                    | Да                               |
| Eval queries                       | Да, `data/processed/eval_queries.jsonl` | DS                                    | Да                               |
| Qrels                              | Да, `data/processed/qrels.tsv`          | DS                                    | Да                               |

- Бейзлайн:
  - используем MedQuAD как основной QA-источник;
  - удаляем пустые question/answer;
  - нормализуем и дедуплицируем `question_id`;
  - создаем детерменированный split на уровне `question_id`, чтобы избежать пролива данных между train/val/test;
  - в baseline `corpus.jsonl` строится из ответов.
  - на текущий момент уже подготовлены:
    - `16 407` QA-пар;
    - `16 407` retrieval-документов;
    - split `train=13 125`, `val=1 640`, `test=1 642`;
    - `300` eval-запросов и `300` qrels.
- Основной MVP:
  - подмешиваем PubMed и StatPearls как источники дополнительных негативов и retrieval-документов;
  - переходим от answer-as-document к более реалистичному corpus/chunk представлению;
  - проверяем баланс типов запросов и длин документов;
  - проводим EDA на предмет дублей, слишком длинных ответов, искажения тем и проблем с доменной репрезентативностью.
- Формирование выборок:
  - train/val/test фиксируются один раз и далее используются во всех экспериментах;
  - eval benchmark формируется только из test-части;
  - стартовый размер eval-набора: 300 запросов.
- Горизонт и гранулярность:
  - горизонт прогноза не используется, так как задача не временным прогнозированием;
  - гранулярность оценки: query-document и query-answer.
- Целевая переменная:
  - для оценки retrieval релевантен документ, соответствующий правильному answer-doc для данного вопроса;
  - для реранкера целевая переменная бинарная: релевантен/нерелевантен.
- Метрики этапа:
  - полнота и корректность артефактов;
  - отсутствие leakage по `question_id`;
  - ручная sanity-check выборка по 30-50 примерам.
- Необходимый результат этапа:
  - зафиксированные артефакты `qa.jsonl`, `corpus.jsonl`, `splits.json`, `eval_queries.jsonl`, `qrels.tsv`.
- Риски:
  - MedQuAD имеет дубли и неоднородное качество ответов;
  - answer-as-document упрощает retrieval-задачу;
  - PubMed/StatPearls могут слабо совпадать по стилю с пользовательскими вопросами.
- Бизнес-проверка:
  - ручной просмотр набора representative examples;
  - подтверждение, что benchmark отражает предполагаемые сценарии использования.

#### 2.3.2. Этап 2. Построение baseline retrieval

Цель этапа: получить нижнюю границу качества, относительно которой будет оцениваться реранкер и более сложный retrieval.

- Бейзлайн:
  - retriever: BM25 по `corpus.jsonl`;
  - index строится один раз и сохраняется как артефакт;
  - evaluation проводится на `eval_queries.jsonl` с `qrels.tsv`;
  - текущие baseline-метрики на 300 hold-out запросах:
    - `P@5 = 0.117`
    - `P@10 = 0.071`
    - `R@5 = 0.587`
    - `R@10 = 0.713`
    - `NDCG@5 = 0.421`
    - `NDCG@10 = 0.462`
- Основной MVP:
  - добавляем dense retriever и hybrid retriever как кандидатогенераторы;
  - сравниваем BM25, dense и hybrid по `Recall@k` и `NDCG@k`;
  - выбираем retriever, который максимизирует вероятность попадания релевантного документа в candidate pool реранкера.
- Формирование выборки:
  - train не нужен для BM25 baseline;
  - валидация и сравнение проводятся на фиксированном hold-out наборе;
  - top-k для benchmark: 5 и 10, candidate pool для реранкера: 20-50.
- Метрики качества и связь с бизнесом:
  - `Recall@k` показывает, насколько часто система вообще находит нужный источник;
  - `NDCG@k` показывает, насколько рано релевантный документ попадает в выдачу, а значит, насколько выше шанс получить grounded answer;
  - `Precision@k` полезен для оценки шумности контекста, который пойдет в generator.
- Необходимый результат этапа:
  - стабильный retrieval baseline и воспроизводимый набор метрик;
  - артефакты индекса и run-файлы для сравнения.
- Риски:
  - BM25 может переоценивать лексическое совпадение и проигрывать на синонимах;
  - dense/hybrid может быть недоступен при отсутствии зависимостей или артефактов;
  - answer-as-document может дать слишком оптимистичный baseline.
- Бизнес-проверка:
  - ручная оценка top-5 документов на части запросов;
  - проверка, что retrieved-контекст действительно пригоден для составления ответа.

#### 2.3.3. Этап 3. Обучение нейросетевого реранкера

Цель этапа: улучшить качество ранжирования по сравнению с baseline retriever.

- Бейзлайн:
  - реранкер отсутствует;
  - документы упорядочиваются score retriever.
- Основной MVP:
  - модель: `PubMedBERT` или близкая biomedical encoder-модель в архитектуре Cross-Encoder;
  - положительные примеры: `(query, true answer/document)`;
  - отрицательные примеры:
    - ответы на другие вопросы;
    - PubMed/StatPearls отрывки;
    - далее в развитии: BM25 top-results без пересечения с позитивом как hard negatives;
  - loss: бинарная классификация релевантности;
  - inference: переоценка top-20/50 кандидатов после retriever.
- Формирование выборки:
  - split строго по `question_id`;
  - validation на отдельном val-сплите;
  - test только для финального сравнения с baseline.
- Гранулярность и частота пересчета:
  - единица обучения: query-document pair;
  - retrain выполняется при существенном изменении корпуса или после накопления новых hard negatives;
  - в пилоте допускается переобучение итерациями вручную.
- Целевая переменная:
  - бинарная релевантность документа для ответа на вопрос.
- Метрики качества и связь с бизнесом:
  - `AUROC`, `F1`, `Accuracy` для pairwise discrimination;
  - `NDCG@10` и `MRR@10` на retrieval benchmark как ключевые ranking-метрики;
  - успех связан с ростом качества top-документов, а значит, с лучшим grounded answer.
- Необходимый результат этапа:
  - реранкер дает улучшение `NDCG@10` минимум на 10% относительно BM25 baseline;
  - при этом `Recall@10` не деградирует более чем на 2 процентных пункта;
  - получен reproducible checkpoint и таблица сравнительных метрик.
- Риски:
  - случайные negatives слишком легкие;
  - domain mismatch между QA-ответами и PubMed abstracts;
  - overfitting на MedQuAD формулировки;
  - дорогой inference при большом candidate pool.
- План управления рисками:
  - добавить hard-negative mining;
  - ограничить пул кандидатов размером 20-50;
  - проводить исследование ошибок по типам;
  - при необходимости заменить модель на более компактную biomedical encoder-модель.
- Бизнес-проверка:
  - pairwise blind comparison top-5 до и после реранкера;
  - ручная проверка, что после реранка документы стали более полезны для генерации ответа.

#### 2.3.4. Этап 4. Генерация ответа и цитирование источников

Цель этапа: превратить retrieval/reranking в пользовательский ответ, сохранив traceability.

- Бейзлайн:
  - локальная small LLM;
  - генерация по top-5 документам BM25;
  - prompt требует отвечать только по контексту и использовать ссылки вида `[doc_id]`.
- Основной MVP:
  - генерация по top-3 или top-5 документам после реранкера;
  - добавляется пост-проверка цитат: цитата считается валидной, если `doc_id` присутствует среди retrieved/reranked документов;
  - при недостаточности контекста система должна явно воздержаться от ответа.
- Формирование выборки:
  - качественная оценка на 50–100 отложенных запросах;
  - отдельный срез сложных вопросов: короткие запросы, редкие термины, длинные multi-fact questions.
- Метрики качества:
  - answer relevance;
  - groundedness / factuality;
  - точность цитирования;
  - доля ответов с корректным отказом при нехватке данных;
  - latency и стоимость inference.
- Связь с бизнес-результатом:
  - retrieval без хорошего слоя формирования ответа неудобен для пользователя;
  - генерация без grounding опасна для медицинского домена;
  - ключевой критерий здесь: удобный ответ без потери проверяемости.
- Необходимый результат этапа:
  - ответ содержит полезное обощение и корректные ссылки на источники;
  - критические галлюцинации отсутствуют на пилотной выборке.
- Риски:
  - LLM может игнорировать инструкции и не выводить цитаты;
  - может копировать нерелевантный фрагмент;
  - длинный контекст ухудшает качество и latency.
- План управления рисками:
  - ограничить число документов и длину контекста;
  - внедрить post-validation источников;
  - использовать deterministic generation в benchmark-режиме;
  - при необходимости перейти к extractive/templated answer для части запросов.
- Бизнес-проверка:
  - ручная оценка ответов по рубрике `полезность / корректность / проверяемость`.

#### 2.3.5. Этап 5. Оценка качества и подготовка решения к пилоту

Цель этапа: выбрать конфигурацию, которая пойдет в пилот.

- Бейзлайн:
  - сравниваем BM25-only retrieval и BM25 + generation;
  - фиксируем исходные retrieval-метрики и qualitative behavior generator.
- Основной MVP:
  - сравниваем минимум три конфигурации:
    - BM25-only;
    - best retriever without reranker;
    - best retriever + cross-encoder reranker + generation;
  - проводим анализ ошибок:
    - релевантный документ не найден;
    - найден, но плохо ранжирован;
    - найден и ранжирован, но генератор ошибся;
    - цитирование не совпадает с фактически использованным контекстом.
- Формирование выборки:
  - фиксированный benchmark на 300 запросов;
  - экспертный набор на 50-100 вопросов для ручной оценки.
- Метрики этапа:
  - retrieval: `P@k`, `Recall@k`, `NDCG@k`;
  - reranker: `AUROC`, `F1`, `Accuracy`, `NDCG@k`;
  - generation: groundedness, citation precision, latency.
- Необходимый результат этапа:
  - финальный отчет по экспериментам;
  - выбор конфигурации для пилота и сервиса;
  - список ограничений и рекомендаций по следующей итерации.
- Риски:
  - offline метрики могут не полностью отражать пользовательскую ценность;
  - benchmark может быть слишком близок к training data распределению.
- Бизнес-проверка:
  - согласование пилотного сценария и критериев успеха;

#### 2.3.6. Этап 6. Сервисный контур MVP

Цель этапа: упаковать выбранную конфигурацию в воспроизводимый сервис.

- Бейзлайн:
  - CLI и локальный single-process запуск достаточны для экспериментов.
- Основной MVP:
  - FastAPI для submit/status/result запросов;
  - Celery worker для асинхронного inference;
  - RabbitMQ как broker;
  - Postgres или file storage для job/result metadata;
  - Docker Compose для разворачивания `app + worker + rabbitmq + postgres`.
- Формирование нагрузки:
  - сервис оптимизирован под demo/pilot, а не под high-load;
  - ожидаемый режим: единичные запросы и короткие очереди.
- Метрики этапа:
  - end-to-end latency;
  - success rate job processing;
  - число ошибок по типам.
- Необходимый результат этапа:
  - работающий API с задокументированными API;
  - воспроизводимый локальный запуск;
  - возможность демонстрации полного пути от вопроса до ответа.
- Риски:
  - зависимость от локального кэша моделей;
  - отсутствие production-grade monitoring;
  - деградация latency на CPU-only окружении.
- Бизнес-проверка:
  - end-to-end smoke test;
  - демонстрация нескольких типовых сценариев для пилота.

### 3. Подготовка пилота

#### 3.1. Способ оценки пилота

- Дизайн пилота двухступенчатый:
  - `offline pilot`: сравнение baseline и MVP на фиксированном benchmark;
  - `expert demo pilot`: слепая ручная оценка ответов на ограниченной выборке вопросов.
- Offline часть:
  - benchmark на 300 запросах;
  - сравнение retrieval и reranking по `P@k`, `Recall@k`, `NDCG@k`;
  - сравнение latency и стабильности inference.
- Expert demo pilot:
  - 50-100 вопросов;
  - оценка по шкалам:
    - релевантность ответа;
    - фактологическая корректность;
    - полезность для пользователя;
    - корректность ссылок на источники.
- Формат сравнения:
  - слепая попарная оценка `baseline vs MVP`;
  - фиксация кейсов, где реранкер улучшил retrieval, и кейсов, где ошибка произошла на стадии generation.

#### 3.2. Что считаем успешным пилотом

- Формализованные критерии:
  - `NDCG@10` у MVP выше baseline минимум на 10%;
  - `Recall@10` не хуже baseline более чем на 2 п.п.;
  - не менее 80% ответов на экспертной выборке оцениваются как полезные и не содержащие критических фактических ошибок;
  - precision корректных citations не ниже 90% на экспертной выборке;
  - сервис успешно завершает не менее 95% пилотных запросов без необработанных ошибок.
- Дополнительные критерии:
  - есть понятная карта ошибок;
  - есть обоснованный выбор между BM25-only, hybrid и RAG+reranker;
  - стоимость и latency укладываются в рамки MVP.

#### 3.3. Подготовка пилота

- Ограничения по вычислениям:
  - baseline BM25 дешев и может работать на CPU;
  - реранкер дает основную вычислительную нагрузку, поэтому candidate pool должен быть ограничен;
  - для первой итерации используется компактная локальная LLM, чтобы не завязываться на внешние API.
- План оценки вычислительной сложности:
  - измерить latency retrieval, reranking и generation отдельно;
  - измерить end-to-end latency на 1 запрос и на короткой серии запросов;
  - при превышении лимитов уменьшить `top_k`, размер candidate pool или размер генеративной модели.
- Ограничения для пилота:
  - асинхронный режим через job API;
  - single-host deployment;
  - ограниченный набор representative use cases вместо широкого внешнего запуска.

### 4. Внедрение

Раздел включен, потому что итогом итерации является не только исследовательский benchmark, но и MVP-сервис.

#### 4.1. Архитектура решения

https://drive.google.com/file/d/10CKOZmT12w3f8RrQH5v6khUunYM98vQ2/view?usp=sharing

- Основные сервисы:
  - API-сервис принимает запросы и отдает статус/result;
  - worker выполняет retrieval, reranking и generation;
  - broker развязывает прием запросов и тяжелый inference;
  - storage хранит jobs и results;
  - MLflow хранит историю экспериментов;
  - артефакты индексов и моделей лежат в файловых volume.
- Методы API:
  - `GET /health`
  - `POST /jobs`
  - `GET /jobs/{job_id}`
  - `GET /jobs/{job_id}/result`

#### 4.2. Описание инфраструктуры и масштабируемости

- Выбранная инфраструктура:
  - локальный Docker Compose;
  - FastAPI + Celery + RabbitMQ + Postgres;
  - volume-based storage для данных, кэша моделей и артефактов.
- Почему выбрана именно она:
  - минимальная операционная сложность;
  - воспроизводимость для ВКР и демонстрации;
  - легко запускать локально и на одном сервере;
  - уже соответствует текущему состоянию проекта.
- Плюсы:
  - быстрый старт;
  - понятная отладка;
  - можно масштабировать worker отдельно от API;
  - не нужен тяжелый orchestration layer.
- Минусы:
  - single-host ограничение;
  - нет полноценной отказоустойчивости;
  - ручное управление секретами и конфигурацией;
  - слабее production observability.
- Почему лучше альтернатив на текущем этапе:
  - Kubernetes/minikube для MVP избыточны;
  - Чистые notebook/CLI не подходят для пилотного API-сценария;
  - полностью умправялемый cloud усложняет воспроизводимость и повышает стоимость.

#### 4.3. Требования к работе системы

- Для MVP не задается production SLA уровня 99.9%.
- Целевые требования пилота:
  - доступность сервиса в демонстрационном окне: не ниже 95%;
  - `P50` latency для BM25-only режима: до 3 секунд;
  - `P95` latency для full pipeline с реранкером и generation: до 8 секунд в целевом pilot-окружении;
  - пропускная способность: 1-3 RPS или до 100-300 запросов в сутки;
  - отсутствие накопления очереди дольше 30 секунд при типовой нагрузке.

#### 4.4. Безопасность системы

- Потенциальные уязвимости:
  - prompt injection через текст retrieved-документов;
  - небезопасные или вредоносные пользовательские запросы;
  - перегрузка очереди слишком длинными запросами;
  - подмена model artifacts;
  - уязвимости сторонних Python-зависимостей.
- План снижения рисков:
  - ограничение длины входного вопроса;
  - deterministic prompts и строгие инструкции отвечать только по контексту;
  - фиксированные проверенные model artifacts;
  - контейнеризация и изоляция сервисов;
  - rate limiting при внешнем доступе;
  - регулярная проверка зависимостей.

#### 4.5. Безопасность данных

- В текущем MVP используются открытые публичные медицинские корпуса.
- Персональные данные пациентов, clinical notes и EHR-данные не входят в текущий скоуп.
- Логи запросов и результатов должны храниться ограниченно и не содержать PII.
- Если в будущем в систему будут подаваться реальные пользовательские медицинские кейсы:
  - потребуется отдельная legal/privacy оценка;
  - потребуется политика retention;
  - потребуется анонимизация и разграничение доступа;
  - нужно будет отдельно проверить соответствие GDPR и локальным требованиям по медданным.

#### 4.6. Издержки

- Для текущего локального MVP прямые облачные издержки могут быть нулевыми, если система запускается на уже доступной машине.
- Основные драйверы стоимости:
  - хранение артефактов и корпусов;
  - inference cross-encoder;
  - модель генерации;
  - message broker и metadata storage.

| Сценарий              | Состав                                             | Оценка издержек                                      |
| --------------------- | -------------------------------------------------- | ---------------------------------------------------- |
| Локальный MVP         | локальная машина, Docker Compose, публичные данные | прямые внешние расходы близки к 0                    |
| Single-host CPU pilot | 1 сервер CPU + RAM + disk                          | низкие, подходят для demo и benchmarking             |
| Single-host GPU pilot | 1 сервер + GPU для reranker/generation             | средние, нужны при росте latency или объема запросов |

- Для выбора конфигурации в пилоте сначала измеряется фактическая стоимость одного запроса:
  - retrieval cost;
  - reranker cost;
  - generation cost.
- После baseline-эксперимента лимиты пересматриваются. Если full pipeline слишком дорогой, fallback-сценарий:
  - BM25-only;
  - BM25 + reranker без generation;
  - уменьшение размера generator.

#### 4.7. Integration points

- Внешние точки интеграции:
  - UI или CLI вызывает `POST /jobs`;
  - клиент опрашивает `GET /jobs/{job_id}` и `GET /jobs/{job_id}/result`.
- Внутренние точки интеграции:
  - preprocessing pipeline подготавливает benchmark-артефакты;
  - indexing pipeline строит retrieval-индексы;
  - training pipeline обучает и сохраняет реранкер;
  - inference worker использует retrieval index, checkpoint реранкера и генеративную модель.
- Интеграция между сервисами:
  - API пишет job metadata в storage;
  - broker передает задачу worker;
  - worker сохраняет итоговый payload в result storage.

#### 4.8. Риски

- Основные риски проекта:
  - retrieval benchmark окажется слишком простым из-за answer-as-document корпуса;
  - реранкер не даст значимого улучшения над сильным BM25 baseline;
  - dense/hybrid контур усложнит систему, но не даст нужного прироста;
  - генерация будет нестабильна и без надежных citations;
  - latency full pipeline окажется слишком высокой для пользовательского сценария;
  - юридические ограничения вырастут при переходе на реальные пользовательские медицинские запросы.
- План работы с рисками:
  - фиксировать baseline и сравнивать только с ним;
  - проводить регулярный error analysis;
  - отдельно оценивать retrieval, reranking и generation, а не только итоговый ответ;
  - держать fallback на более простой pipeline;
  - не расширять скоуп до production use case без отдельного design doc следующей итерации.
